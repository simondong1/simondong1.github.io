Understanding MLA (Multi-head Latent Attention)

Why DeepSeek-style models store one latent KV state, why the head splits into nope and rope, and why training code looks different from optimized decode.

Simon Dong•July 2026•~14 min read

MLA is easiest to understand as a compromise between MHA and MQA. It stores one shared latent per token like MQA stores one shared KV head, but it still gives each query head its own learned view of that latent. The result is MQA-like cache size with much more MHA-like expressiveness.

Where MLA is used

MLA is the attention family behind DeepSeek-V2/V3 and the Kimi K2.x text backbone, and MLA-style low-rank KV designs also show up in newer 2026 systems such as GLM-5.2 and Mistral 4. In this post, I use DeepSeek-V3 dimensions and code paths as the concrete reference.

Series context

This post follows the GQA post. GQA saves cache by reducing the number of KV heads. MLA goes further: it replaces explicit cached K/V heads with a compact latent representation, then factors the decode math so serving engines do not rebuild full per-head K/V for every cached token.

1.Where MLA sits: MHA, MQA, then MLA

In standard multi-head attention, every query head owns its own key and value head. That is expressive, but the KV cache is huge.

MHA:  many Q heads, many cached K/V heads
GQA:  many Q heads, fewer cached K/V heads
MQA:  many Q heads, one cached K/V head
MLA:  many Q heads, one cached latent, plus a tiny rope key

The closest storage analogy is MQA: both store one shared thing per token. The difference is that MQA stores one literal K/V head, while MLA stores a richer latent c_kv. Each query head then has its own learned projection from that shared latent, so heads can still see different K/V subspaces.

MQA:  all heads use the same K,V
MLA:  all heads share c_kv, but head h has its own W_K^h and W_V^h

2.What MLA tries to achieve

MLA has two separate goals. The first is storage: cache a tiny per-token representation instead of full per-head K/V. The second is decode compute: avoid rebuilding full per-head K/V for every cached token at every generation step.

GoalWhat happensWhy it matters
Storagecache c_kv + k_ropefar fewer values per token/layer
Decode computemove K/V projection work to current query/outputdo not up-project the whole cache every step

For DeepSeek-V3 dimensions, a full reconstructed cache would be roughly:

K_nope: 128 heads * 128 dims = 16384
K_rope: 128 heads *  64 dims =  8192
V:      128 heads * 128 dims = 16384
full:                         40960 values/token/layer

The optimized MLA cache stores:

c_kv:   512
k_rope: 64
latent cache: 576 values/token/layer

That is about 71× smaller for the cached state. The training/reference implementation may reconstruct full K/V; the optimized decode implementation stores the latent.

3.The two paths: nope and rope

MLA splits Q and K into two parts:

Q = [ Q_nope | Q_rope ]
K = [ K_nope | K_rope ]

Nope means no positional embedding. This is the content path. It can be represented as a fixed linear projection from the latent, so it is cache-compressible and decode-factorable.

Rope carries position. If the notation is unfamiliar, see the earlier RoPE post; the short version is that a key at position n gets a rotation depending on n. Because that rotation changes with key position, this part cannot be treated as one fixed projection from the latent during decode.

NOPE path:  c_kv -> per-head K_nope and V       (fixed learned maps)
ROPE path:  hidden -> small K_rope -> RoPE      (position-dependent)

Only the nope path gets both benefits: compact storage and decode factoring. The rope path still saves storage by being tiny and shared across heads, but it does not get the same decode factoring.

4.The training/reference setup: compress, then reconstruct

The Hugging Face DeepSeek-V3 implementation is the clean reference path. It trains the real MLA weights, but it reconstructs full per-head K/V before attention. That is correct for training and simple generation; it is not the optimized latent-cache decode path.

DeepSeek-V3 dimensions:
hidden_size       = 7168
num_heads         = 128
q_lora_rank       = 1536
kv_lora_rank      = 512
qk_nope_head_dim  = 128
qk_rope_head_dim  = 64
qk_head_dim       = 192
v_head_dim        = 128

The Q path may also use a low-rank projection:

hidden -> q_a_proj(1536) -> RMSNorm -> q_b_proj(128 heads * 192)
q -> split into q_nope(128) + q_rope(64)

The KV path is the core MLA path:

hidden -> kv_a_proj_with_mqa(512 + 64)
       -> split into c_kv(512) and k_rope(64)

c_kv -> RMSNorm -> kv_b_proj -> per-head K_nope(128) and V(128)
k_rope -> RoPE -> shared across heads

The latent RMSNorms are not QK-norm. They normalize the low-rank bottlenecks (q_lora_rank and kv_lora_rank) so the compress/decompress path trains stably.

Important: during standard pretraining or finetuning, there is no autoregressive KV cache. The model sees the full sequence, uses a causal mask, computes full K/V for the forward pass, and backpropagates. So the huge latent-cache saving is a decode-time serving win, not a training-forward win.

5.The optimized decode setup: do not rebuild the cache

Serving engines such as SGLang and vLLM reorganize the same trained weights for decoding. The simplest code to read is SGLang's MiniCPM3 MLA path. It splits the trained kv_b_proj into two matrices:

w_kc: per-head K projection, moved to the query side
w_vc: per-head V projection, moved to the output side

For K, the reference computation is:

K_nope_h[n] = W_K^h c_kv[n]
score_h[n]  = Q_nope_h · K_nope_h[n]

Decode factors this as:

score_h[n] = (Q_nope_h W_K^h) · c_kv[n]

The current query is transformed once into latent space, then dotted against every cached latent. That avoids applying W_K^h to every cached token on every step.

For V, the same idea happens after attention:

reference:  output_h = Σ_n A[n] (W_V^h c_kv[n])
decode:     output_h = W_V^h (Σ_n A[n] c_kv[n])

So the weighted sum happens in latent space, and only the final result is projected back to v_head_dim. The model does not reconstruct V for every cached token during optimized decode.

SGLang decode sketch:
q_nope -> w_kc -> latent query (512)
cache stores [c_kv(512) | k_rope(64)]
attention runs over latent cache
latent output -> w_vc -> per-head output (128)
o_proj -> hidden

6.Why rope stays outside the latent

This is the part that makes MLA feel strange. The nope path uses a fixed matrix W_K^h. It is different for each head, but fixed across all token positions, so we can move it to the query side once for the current decode token.

RoPE is different. At decode position m, the current query attends to cached key positions n = 0, 1, ..., m-1. The relative rotation is R_(m-n), so every cached position gets a different rotation:

n=0:   R_m
n=1:   R_(m-1)
n=2:   R_(m-2)
...

Those rotations are predictable, but they are not one shared matrix. If rope were hidden inside the latent projection, you would need a different transformed query for every cached key position. That destroys the single-query-against-whole-cache trick.

DeepSeek's fix is to keep the rope part small and explicit:

k_rope: 64 dims, one shared head
q_rope: 64 dims, per query head

The per-head content interaction lives in the nope path. The rope path only carries enough positional signal to keep relative position working, so it is tiny and shared across heads.

7.Summary

MLA = MQA-like cache + MHA-like per-head views

NOPE path:
  content path
  stored as c_kv
  can move K/V projection work to query/output side

ROPE path:
  position path
  kept outside c_kv
  tiny and shared across heads
  cannot use the same decode factoring

Transformers implements the readable training/reference form: compress, reconstruct full K/V, attend. SGLang/vLLM implement the optimized decode form: cache c_kv + k_rope, move the K/V projections to the query/output side, and avoid rebuilding full per-head K/V for every cached token. Same weights, different runtime factorization.

MLA in one line: store one latent KV state, let each head learn its own view of it, and keep a tiny separate rope key for position.
