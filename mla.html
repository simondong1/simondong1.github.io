Understanding MLA (Multi-head Latent Attention)

DeepSeek-style models store one small latent per token instead of full keys and values. This post shows what is compressed, why the head splits into a content part and a position part, and why position cannot be compressed.

Simon Dong•July 2026•~12 min read

MLA is a low-rank version of multi-head attention. It keeps every query head, but stores keys and values as one shared latent per token — like Multi-Query Attention's tiny cache, yet each head still gets its own learned view. This post is the architecture; a follow-up covers the two ways to run it at inference.

Where MLA is used

MLA is the attention behind DeepSeek-V2 / V3 and the Kimi K2.x text backbone, and MLA-style low-rank KV also appears in newer 2026 systems such as GLM-5.2 and Mistral 4. Every number here uses DeepSeek-V3 as the concrete reference.

Assumed background

Multi-head attention (Q, K, V, one cache per head) and RoPE. If RoPE is rusty, the RoPE post covers it. Everything else is defined here.

1.Where MLA sits: MHA → MQA → MLA

All four schemes keep many query heads. They differ only in what the KV cache holds per token.

SchemeWhat is cached per tokenPer-head views?Cache size
MHAone full key and value per headyes, independentlargest
GQAkey/value for a few shared groupsshared in groupsmedium
MQAone key and one value, shared by all headsno, identicalsmallest
MLAone latent vector + one small shared position keyyes — reconstructed per headnear-MQA

The key idea: MQA is cheap because it shares one key/value, but every head sees the same thing, which costs quality. MLA also caches one shared object, but it is a richer latent from which each head reconstructs its own key and value with its own learned matrix. So MLA gets MQA-like storage with MHA-like head diversity.

One-line placement: MLA = the small cache of MQA, the per-head expressiveness of MHA.

2.What is actually "MLA" (the common trap)

Reading the code, it is easy to think MLA is "LoRA on Q and K." It is not. Two independent things live in the attention block, and only one of them is MLA.

PieceWhat it isIs it MLA?
Query low-rankoptional parameter compression of the query projectionNo — just saves weights; the query is not cached
KV latentcompress K and V for all heads into one latent, decompress per headYes — this is MLA

The query low-rank path is optional (a model can skip it). The KV latent is the whole point and is always present. So when we say "MLA," we mean: keys and values for every head are generated from one shared, cached latent. Formally, for a token's hidden vector \(x\), one small matrix produces the latent

\[ c_{kv} \;=\; x\,W_{\text{down}}, \qquad c_{kv}\in\mathbb{R}^{\text{latent\_dim}} \]

and then each head \(h\) reconstructs its own key and value from that same latent:

\[ K^{(h)} \;=\; c_{kv}\,W_K^{(h)}, \qquad V^{(h)} \;=\; c_{kv}\,W_V^{(h)}. \]

Only \(c_{kv}\) is cached. The per-head matrices \(W_K^{(h)}, W_V^{(h)}\) are weights, not cache — which is exactly why each head can differ while the storage stays small.

3.The dimensions (DeepSeek-V3), and the constraints

These names look like many free knobs, but several are tied together. Here are the real values and which are independent.

NameMeaningValueFree or derived?
headsnumber of attention heads128free
latent_dimwidth of the cached KV latent512free
nope_dimcontent part of the query/key head128free
rope_dimposition part of the query/key head64free
qk_head_dimfull query/key head = nope + rope192derived: nope + rope
value_dimvalue width per head128free, independent of qk_head_dim

Two constraints trip everyone up:

  • The query/key head is split, not free: qk_head_dim = nope_dim + rope_dim. For DeepSeek-V3, 192 = 128 + 64.
  • Value width is independent and need not equal the query/key width. Here value_dim = 128 but qk_head_dim = 192. This is fine: the score sum runs over qk_head_dim (Q vs K), while the output width is value_dim (weights vs V). They are two different matmuls, so they can have two different widths.

4.The two paths inside a head: nope and rope

Each query and key head is concatenated from two pieces with different jobs:

\[ Q = [\,Q_{\text{nope}} \;\Vert\; Q_{\text{rope}}\,], \qquad K = [\,K_{\text{nope}} \;\Vert\; K_{\text{rope}}\,]. \]
  • nope = "no positional encoding." The content part. It comes straight from the latent via a fixed learned matrix, so it is compressible.
  • rope = the position part. It gets a rotary rotation that depends on the token's position, so it cannot be a fixed function of the latent.

The value has no such split — it is just value_dim, all content. Here is the actual data flow, drawn out:

5.The MLA forward, step by step

Here is the whole forward for a batch of T tokens, in order, with the output shape of each step (DeepSeek-V3 numbers). This is the concrete algorithm the diagram above summarizes — and it is where the latent norm actually lives.

#OperationOutput shape
1down-project the hidden state to the KV latent + shared rope key(T, 512 + 64)
2RMSNorm the latent (the 512 part only)(T, 512)
3up-project the latent to per-head K_nope and V(T, 128 heads, 128 + 128)
4RoPE the 64-wide rope key, then share it across all heads(T, 1→128, 64)
5project the query, split into Q_nope(128) + Q_rope(64), RoPE the rope part(T, 128 heads, 192)
6concat per head: K = [K_nope | rope], Q = [Q_nope | rope](T, 128 heads, 192)
7attention: softmax(Q·Kᵀ / √192) · V(T, 128 heads, 128)
8flatten heads and output-project back to hidden(T, 7168)

Only the width-576 object from step 1 (the 512 latent plus the 64 rope key) is written to the cache. Everything in steps 3–7 is recomputed from it. Three details in this walkthrough are the ones people miss:

Easy to miss

Step 3 — one latent, two roles. The same cached latent is up-projected into both K_nope and V. The latent is not "the key"; it is the shared source of key and value.

Step 4 — the rope key has no per-head weights. It is produced directly in step 1 and just broadcast to all heads (a copy, no matmul). That is why one 64-wide vector serves every head.

Step 2 — the norm is on the latent, not on K or V. It normalizes the compressed bottleneck before step 3 expands it. Because RMSNorm divides by the vector's own magnitude it is nonlinear, so \( W\cdot\operatorname{RMSNorm}(c_{kv}) \neq \operatorname{RMSNorm}(W\,c_{kv}) \) — normalizing the latent is deliberately different from normalizing each expanded head. It is a bottleneck-stability norm, not the per-head QK-norm seen in models like Qwen3.

The reason the cache only needs step 1's output is the whole payoff. Per token, per layer:

MHA-style full cache
40,960 values
MLA latent cache
576 values

That is 512 (latent) + 64 (shared rope key) = 576 stored values, versus 128×(192+128) = 40,960 for a full multi-head cache — about 71× smaller.

6.Why position cannot live in the latent

This is the reason for the whole nope/rope split. The content part works because its reconstruction matrix is fixed: the same \(W_K^{(h)}\) applies to every cached token, so the score can be regrouped. Written out for one head, one query position \(m\), one key position \(n\):

\[ Q_{\text{nope}}\cdot K_{\text{nope}} \;=\; Q_{\text{nope}}\,\big(W_K^{(h)}\,c_{kv}\big) \;=\; \big(Q_{\text{nope}}\,W_K^{(h)}\big)\cdot c_{kv}. \]

The matrix \(W_K^{(h)}\) does not depend on \(n\), so it can be moved onto the query side once (this is what makes efficient decoding possible — Part 2). Position breaks that. RoPE inserts a rotation \(R_{\,m-n}\) that depends on both positions, sitting between query and key:

\[ Q_{\text{rope}}\cdot K_{\text{rope}} \;=\; Q^{\top} R_{\,m-n}\, K. \]

There is no single fixed matrix here — \(R_{\,m-n}\) is different for every cached position \(n\):

key at position 0  → rotate by  R(m-0)
key at position 1  → rotate by  R(m-1)
key at position 2  → rotate by  R(m-2)
...                                (a different rotation per cached token)

So position cannot be folded into the fixed latent reconstruction. DeepSeek's fix is to carry position separately: a small rope_dim = 64 slice, shared across heads, that is rotated per position and concatenated back on. The content lives in the compressible latent; the position rides alongside in a tiny explicit vector.

The clean 2×2 view of the whole scheme:

Small cache?Fixed reconstruction?
nope (content)yes — in the latentyes — fixed \(W_K^{(h)}\)
rope (position)yes — but tiny & shared, not in the latentno — rotation changes per position

7.Training vs decode (a one-paragraph preview)

The same MLA weights are run two different ways. During training (and in the readable reference code), the latent is expanded back into full per-head K and V, and ordinary attention runs — simple and easy to differentiate, no cache. During decode, serving engines keep only the latent in the cache and fold the reconstruction matrices onto the query and output, so they never rebuild full K/V for every cached token. Same math, two factorizations. The trade-off between them — and why both exist — is the subject of Part 2.

8.Summary

QuestionAnswer
What is MLA?compress K and V for all heads into one cached latent; reconstruct per head
Is the query low-rank part MLA?no — that is optional weight compression; MLA is the KV latent
Why the nope/rope split?content compresses into the latent; position cannot, so it rides separately
Is rope per head?no — the rope key is one small shared vector, broadcast to all heads
What does the norm touch?the shared latent bottleneck, not per-head K/V
How big is the win?~71× smaller KV cache for DeepSeek-V3

MLA in one line: cache one latent per token, let every head learn its own key/value from it, and keep a tiny shared rope key because position refuses to compress.
